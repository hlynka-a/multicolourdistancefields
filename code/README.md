# Multi-Colour Distance Fields - Code



Code repository for the project published in the paper

&#x09;*Multi-Colour Distance Fields* (2026)(Hlynka and Mould).



The demo code presented here is written in **C#** and **Unity3D 6000.0.47f1**.

This is presented as a visual demo, with a runtime user interface to dynamically load different images, generate distance fields, and change their render settings.

The scripts most likely relevant to you are

**/Assets/Scripts/df_scripts/DF_Final_DistanceFieldManager.cs**

**/Assets/Shaders/DistanceFieldShaderInteractive.shader**


This demo includes a collection of example images from *OpenClipArt.org*.

#### ANSWERS

[**What Unity3D settings / versions should I use?**](#q03)<a id="q03"></a>

We used Unity3D version 6000.0.47f1, with a 3D template (built-in rendering pipeline), and used the legacy input manager. You may receive warnings related to these features, we recommend ignoring them to attempt running the sample scene.

[**The demo freezes / tales a long time when trying to generate a Distance Field.**](#q01)<a id="q01"></a>

The demo freezes until it completes creating a Distance Field. This may take several seconds, or several minutes, depending on the resolution and complexity of the output. Our method is CPU-based, and as it would normally be an offline task, we did not focus on ways to optimize this task (e.g. calculating the field on the GPU). In our analysis, a significant bottleneck appears to be in the algorithm that solves the 4-Color-Mapping-Problem for RegSDF, which can be found in the function "private float[,] Calculate4ColorMapping(Texture2D ogTex)" in DF_Final_DistanceFieldManager.cs. The second-worst bottleneck is our calculation of RelSDF, which can be found in the function "private float[,] CalculateTrueSignedDistanceFieldLayer(float[,,] ogTex, Color outlineColor, float colorTolerance, int neighborhoodRange)" in DF_Final_DistanceFieldManager.cs.

[**The demo sometimes fails to generate a Distance Field.**](#q02)<a id="q02"></a>

There are several points in the pipeline where failure can occur for a limited subset of images. RelSDF and RegSDF are both generated by default; try toggling these to only generate RelSDF or only RegSDF to see if this helps, or try generating a standard SDF or DF rather than our version.

[**Do I need to generate the Distance Field data during runtime, every time?**](#q04)<a id="q04"></a>

The demo recreates the Distance Fields on demand every time, but this isn't necessary. The typical pipeline would be to generate your Distance Field data offline, save the data to lossless RGBA files (e.g. .png), and reload them back into a new scene at runtime to immediately render them. Keep in mind that the A value in RGBA values may be treated differently from R, G, or B by Unity3D, and avoid turning on any compression options in the game engine. See the Supplementary Materials .pdf for more details. Some code to save the distance field data to file to reload elsewhere is already present in DF_Final_UI_Manager.cs. 

[**My results don't look like what was shown in the paper?**](#q05)<a id="q05"></a>

Different settings might cause different results in rendering quality, the amount of colours in the resulting distance field, etc. To adjust your settings, try using the user interface during runtime, or look for the corresponding parameters in Unity3D's insepctor panel (or as a last resort, dig through variable values used in the code). Quick options include disabling RelSDF and only generating RegSDF (especially if image does not have an obvious outline or single colour for an outline), and adjusting the Color Tolerance value that determines how similar colours must be to be considered identical during colour matching (see _ColorTolerance in DistanceFieldShaderInteractive.shader).
